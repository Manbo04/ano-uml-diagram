📚 How to Read This UML Class Diagram

A UML Class Diagram shows the structure of a system — its tables (classes), their attributes (columns), and how they relate to each other. Here's a quick visual guide to every concept used in this diagram:

Class (Table)

Each box represents a database table. The header shows the table name, and the stereotype (in «angle brackets») tells you its role: entity, dictionary, mapping, legacy, or system.

Attributes (Columns)

Each line inside a class box is a column. They're color-coded:

Red = Primary Key (PK) — unique identifier for each row

Blue = Foreign Key (FK) — references another table's PK

Green = Index — speeds up queries on this column

Gray = Regular column — normal data field

Association (Relationship)

A line between two tables means they're related via a Foreign Key. The relationship table at the bottom shows all connections.

The arrow points from parent to child. "1 → *" means one-to-many.

Multiplicity (Cardinality)

Numbers at each end of a relationship line show how many records can exist:

1 → * — One-to-Many: one user has many provinces

1 → 0..1 — One-to-Zero-or-One: a user belongs to at most one coalition

* → * — Many-to-Many: resolved via a mapping table

Dictionary + Mapping Pattern

The core design pattern used in this database. A dictionary table defines what things exist, and a mapping table tracks how much each user has.

Adding a new resource = INSERT a row. No schema change needed!

Aggregation & Composition

Aggregation (open diamond) means "has a" — the child can exist independently. E.g., a coalition has members, but users exist without coalitions.

Composition (filled diamond) means "owns" — the child can't exist without the parent. E.g., coalition_banks can't exist without a coalition.

Generalization (Inheritance)

An open triangle arrow means "is a type of". In our DB, the dictionary tables act as abstract definitions that are specialized by their mapping tables. Think of it like: user_economy "extends" the concept defined in resource_dictionary.

Color Legend (Migration Status)

Each table is color-coded by its current status in the database migration:

Blue — Active normalized tables (in use)

Red — Legacy tables (to be dropped)

Yellow — Created but never wired in

Green — System tables (unchanged)

Migration Status Notes:

● Active (blue): Dictionary + mapping tables created by migrations 0002–0004. Currently in use by province.py, countries.py, action_loop.py.
● Legacy (red): Original denormalized wide tables. Still actively queried by coalitions.py (100%), market.py (100%), and partially by tasks.py. To be dropped after Phase 3.
● Unused (yellow): Tables that were created by migrations 0003 but never wired into application code. Need data migration + coalitions.py rewrite in Phase 1.
● System (green): Infrastructure tables that remain unchanged across all migration phases.

Key pattern: Every game entity follows the Dictionary + Mapping pattern. The dictionary table defines what exists (e.g., "oil" in resource_dictionary). The mapping table tracks player quantities (e.g., user 42 has 5,000 oil in user_economy). This means adding a new resource is an INSERT, not an ALTER TABLE.
